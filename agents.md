# Agent Working Rules

## 项目执行基线

本文件是后续实现阶段的轻量执行规范。目标是让每次开发都能保持架构一致、测试闭环和文档同步，不引入沉重流程。

必须遵守的项目基线：

- 客户端主路线固定为 Flutter，覆盖 Android 和 Web。
- 服务端主路线固定为 Go 单二进制 + SQLite。
- Web 支持手动新增交易，但不支持短信扫描。
- Android 支持短信扫描，私有安装包分发，不上架 Google Play。
- 系统不支持离线新增或离线编辑。
- 生产公网访问使用 HTTPS + 一次性配对码 + 设备 token，不设计密码登录。
- 金额持久化使用 `amount_cent INTEGER`，单位为 0.01 RMB。
- 照片最终压缩由服务端 FFmpeg 完成。
- 删除采用软删除。
- 备份迁移必须同时考虑 Windows 与 Ubuntu Linux。
- `scripts/ubuntu/` 下的安装、启停、数据库迁入迁出脚本属于发布运维接口；后续更新应尽可能保持脚本文件名、参数和环境变量兼容，除非安全或数据一致性要求必须变更。

如果实现需求与上述基线冲突，必须先更新 `docs/` 中对应设计并向用户说明影响，再继续实现。

## 文档同步规则

代码实现必须和 `docs/` 保持一致。

- 新增或改变功能边界时，先更新对应设计文档或 TODO。
- API 请求/响应变化时，更新 `docs/architecture/api-security.md` 或相关服务端模块文档。
- 数据库字段、索引、删除策略变化时，更新 `docs/architecture/data-model.md`。
- Flutter 平台差异变化时，更新 `docs/client/flutter-architecture.md` 或对应客户端模块文档。
- 备份、迁移、FFmpeg、跨平台路径变化时，更新部署或服务端模块文档。
- 完成功能后，更新 `docs/TODO.md` 中对应任务状态或追加下一步任务。

文档不需要长篇解释，但必须让下一位实现者不需要重新猜决策。

## 技术栈约束

默认不得引入以下必需依赖：

- Docker。
- Nginx。
- PostgreSQL。
- Redis。
- 原生 Android 客户端替代 Flutter。
- Vue/React Web 客户端替代 Flutter Web。

允许作为可选部署辅助的内容必须写清楚“非必需”。例如公网环境可以使用外部反向代理或云厂商 HTTPS，但系统本身不能依赖它才能本地运行。

新增第三方库前必须满足：

- 明确解决当前功能问题。
- 不破坏 Windows 与 Ubuntu Linux 迁移目标。
- 不显著增加小 CPU 服务器负担。
- 在文档或最终汇报中说明用途。

## 数据与安全边界

- 不记录、上传或持久化短信原文到服务端。
- 日志不得包含设备 token、配对码、短信原文、完整银行卡号或敏感账户标识。
- token 明文只返回给客户端一次，服务端只保存 hash。
- 生产公网模式必须假设网络不可信，所有业务 API 都需要认证。
- 任何备份包都应包含恢复所需的 manifest，并避免写入平台绝对路径。
- SQLite 迁移不得只复制 `app.db`，必须考虑 WAL/checkpoint 或使用备份流程。

## 实现切片规则

每次实现只选一个最小可测功能切片。推荐切片示例：

- 设备配对 API。
- Flutter 配对页。
- 交易新增 API。
- Flutter 交易表单新增。
- 交易列表过滤。
- 附件上传 API。
- FFmpeg 压缩。
- Android 短信手动扫描。

不要在一个未验证切片上继续堆叠多个新功能。若必须跨端同时修改，应仍以一个用户可验证行为作为切片边界。

## 阻塞处理

遇到阻塞时，先尝试用本地环境确认原因。以下情况必须停止继续堆叠实现并向用户说明：

- 本机服务端无法启动。
- Web 闭环测试无法完成。
- Android 模拟器无法启动或无法连接本机服务。
- FFmpeg 不可用且当前功能依赖照片压缩。
- SQLite 迁移或备份流程存在数据丢失风险。
- 权限策略导致短信功能无法验证。

说明阻塞时必须包含：

- 已完成内容。
- 阻塞命令或操作。
- 错误现象。
- 已尝试的排查。
- 建议用户如何选择或提供什么信息。

## 本地最小化闭环测试环境

本项目必须搭建并维护一套本地最小化测试环境，用于在每个功能实现完成后立即验证真实读写链路。

### 本机服务器

- 以本机作为中心服务器启动 Go 服务端。
- 默认测试地址使用本机局域网或 loopback 地址，例如 `http://127.0.0.1:<port>`。
- 服务端必须连接本地 SQLite 测试数据库和本地附件目录。
- 测试环境必须覆盖：
  - 设备配对。
  - 交易新增。
  - 交易读取。
  - 交易编辑。
  - 软删除。
  - 附件上传与服务端 FFmpeg 压缩。
  - 统计接口。
  - 备份与 checkpoint 基础流程。

### Web 闭环测试

- 启动本机服务端后，通过 Flutter Web 入口访问页面。
- 验证 Web 能正常完成：
  - 设备配对或已有 token 登录。
  - 新增交易。
  - 列表读取与过滤。
  - 编辑交易。
  - 删除交易。
  - 上传照片。
  - 查看统计图表。
- Web 不得出现短信扫描入口。

### Android 模拟器闭环测试

- 本机已安装 Android Studio，可以使用 Android Emulator 进行测试。
- 每次涉及客户端关键功能时，应启动模拟器安装或运行 Flutter Android App。
- 模拟器访问本机服务端时使用 Android emulator host 地址，例如 `http://10.0.2.2:<port>`，除非实际网络配置要求使用局域网 IP。
- 验证 Android App 能正常完成：
  - 设备配对或已有 token 登录。
  - 新增交易。
  - 列表读取与过滤。
  - 编辑交易。
  - 删除交易。
  - 上传照片。
  - 查看统计图表。
  - Android 专属短信扫描入口存在。
- 涉及短信模块时，必须验证：
  - 权限申请流程。
  - 后台监听或模拟短信输入流程。
  - 断网时不处理新短信。
  - 联网后用户手动触发重新扫描。
  - 提交服务端时不上传短信原文。

## 实现与测试交替循环

开发过程必须遵循“小步实现、立即测试、确认后继续”的循环。

1. 选择一个最小功能切片。
2. 实现服务端、客户端或共享模型的必要改动。
3. 启动本机服务端。
4. 执行对应的 Web 闭环测试。
5. 如功能涉及 Android 或平台能力，执行 Android 模拟器闭环测试。
6. 修复发现的问题。
7. 重复测试直到该功能切片通过。
8. 记录已验证内容和仍未覆盖的测试点。
9. 只有当前功能切片测试通过后，才继续实现下一个功能切片。

禁止在多个未验证功能上连续堆叠实现。若某个功能无法完成本地闭环测试，必须明确记录阻塞原因、已验证范围和剩余风险。

## 功能完成定义

一个功能只有同时满足以下条件才算完成：

- 服务端 API 或客户端 UI 行为符合 `docs/` 中对应模块设计。
- 本机服务端可以启动。
- Web 端相关路径通过本地读写测试。
- Android 端相关路径通过模拟器测试。
- 数据库写入符合数据模型，金额使用 `amount_cent INTEGER`。
- 若涉及照片，服务端 FFmpeg 压缩结果可访问。
- 若涉及短信，短信原文不上传服务端。
- 若涉及备份迁移，Windows 与 Ubuntu Linux 兼容要求已在实现或测试记录中确认。

## 测试记录要求

每次完成一个功能切片后，最终汇报必须包含：

- 本次实现的功能切片。
- 本机服务端启动方式和访问地址。
- Web 测试结果。
- Android 模拟器测试结果。
- 未测试项及原因。
- 发现并修复的问题。
