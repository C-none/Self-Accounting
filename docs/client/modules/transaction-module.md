# 客户端交易模块设计

## 功能范围

Android 和 Web 均支持：

- 手动新增交易。
- 编辑已有交易。
- 软删除交易。
- 交易列表过滤。
- 交易详情查看。
- 照片附件上传。

Web 不支持短信扫描，但 Web 新增交易的能力与 Android 手动新增一致。

## 交易表单字段

- 金额：用户输入 RMB 小数，提交前转换为 `amount_cent`。
- 方向：收入、支出、转账。
- 时间：交易发生时间。
- 一级分类。
- 二级分类。
- 使用人。
- 账户，展示为银行名称和可选银行卡尾号。
- 交易对象。
- 详细描述。
- 照片附件。

表单性能要求：

- 分类、使用人、账户来自 `/api/bootstrap`，不在每次打开表单时重复拉取。
- 金额转换使用字符串解析，避免浮点误差。
- 提交按钮只在必填字段合法且网络可用时启用。
- 新增和编辑共用同一表单组件，避免重复逻辑。

金额转换：

- 输入：`12.34`
- 请求：`amount_cent = 1234`
- 展示：`amount_cent / 100 = 12.34`

## 网络要求

所有写操作必须在线：

- 新增：调用 `POST /api/transactions`。
- 编辑：调用 `PATCH /api/transactions/{id}`。
- 删除：调用 `DELETE /api/transactions/{id}`。
- 上传照片：调用 `POST /api/attachments`。

无网络时：

- 禁止提交。
- 保留表单当前输入。
- 显示“当前无网络，请联网后重试”。
- 不写本地待同步队列。

## 列表过滤

过滤条件：

- 时间范围。
- 一级分类。
- 二级分类。
- 使用人。
- 账户，按银行名称和可选银行卡尾号展示。
- 收入/支出/转账方向。
- 关键词。

客户端调用 `GET /api/transactions`，服务端负责分页和过滤。

列表性能要求：

- 默认第一页 50 条。
- 下拉或分页加载下一页。
- 附件只显示缩略图，不加载原图。
- 过滤条件变化后重新请求第一页。
- 列表项不使用复杂阴影、动画或嵌套卡片。
- 默认时间范围为结束日期今天、起始日期一周前；输入框为空时以灰色 `YYYYMMDD` 提示默认值。
- 日期过滤支持直接输入 8 位纯数字 `YYYYMMDD`，也支持点击箭头打开日期选择器。
- 交易列表按交易发生日期分组，每天之间显示日期分界线。
- 日期分界线带折叠按钮，可折叠或展开当天全部交易。

## 最后写入胜出

客户端不做复杂冲突处理。编辑提交成功后，以服务端返回交易为准刷新 UI。若多个设备同时编辑同一交易，最后被服务端接收的提交成为最终状态。

## 软删除

删除交易时调用 `DELETE /api/transactions/{id}`。成功后列表隐藏该交易。服务端只设置 `deleted_at`，不立即物理删除。

## 验收标准

- Android 和 Web 都能新增交易。
- Web 页面没有短信扫描入口。
- 金额请求字段始终是整数 `amount_cent`。
- 无网络时不会产生本地待同步数据。
- 删除后交易从默认列表消失，数据库保留软删除记录。
- Web 和 Android 都完成新增、过滤、编辑、删除的闭环测试。
